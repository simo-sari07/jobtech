Project Context & Overview

Background

JobTech Solutions is an enterprise-grade HR platform built for EMSI that digitalises and centralises the complete recruitment lifecycle. The system replaces fragmented manual processes — scattered emails, spreadsheets, and paper notes — with a single unified web application.

The platform manages everything from job posting creation to CV submission, candidate screening, interview scheduling, evaluation, AI-powered scoring, and PDF report generation.

Problem Statement — Why this system is needed
  • Interview management was manual, inconsistent, and untracked across teams
  • No centralised history of past interviews or candidate evaluations
  • Reminders and notifications required manual effort, causing missed sessions
  • No standardised evaluation form — every recruiter used different criteria
  • CV screening was time-consuming with no AI assistance or scoring
  • Reports and statistics required hours of manual spreadsheet work

Goals

  1. Standardise and centralise all interview and recruitment management
  2. Automate scheduling, reminders, and status notifications
  3. Provide structured evaluation with scoring and comparison
  4. Maintain a complete, searchable history of all interviews
  5. Integrate AI for CV parsing, candidate matching, and predictive ranking
  6. Generate exportable PDF and Excel reports with live statistics

Stakeholders & Roles

Complete Use Cases

Authentication

UC-01 User Login All roles ▼
Actors
All roles
Preconditions
User has a registered account and is not logged in
Main Flow
1. User submits email + password 2. System validates credentials 3. System issues JWT access token (15min) + refresh token (7d) 4. Frontend stores tokens in memory / httpOnly cookie 5. User redirected to role-specific dashboard
Postconditions
User authenticated. JWT tokens issued.
Exceptions
Invalid credentials → 401 generic message. Account disabled → 403.
UC-02 Register as Candidate Public ▼
Actors
Candidate (unauthenticated)
Preconditions
User has no existing account
Main Flow
1. User fills: first name, last name, email, password 2. System validates email uniqueness and password strength 3. System creates user with role=CANDIDATE 4. Celery task queued: send welcome email 5. User auto-logged in, redirected to candidate dashboard
Postconditions
New candidate account created. Welcome email queued.
Exceptions
Email exists → 400. Weak password → validation error with strength hint.
UC-03 Token Refresh (Silent) System ▼
Actors
System (axios interceptor)
Preconditions
Access token expired. Refresh token still valid.
Main Flow
1. Axios interceptor catches 401 response 2. Calls POST /api/v1/auth/token/refresh/ automatically 3. System validates refresh token, issues new access token 4. Original failed request retried transparently
Postconditions
New access token issued. Session continues without user action.
Exceptions
Refresh token expired → redirect to login page.

Job Offers

UC-04Create Job Offer Recruiter / HR▼
Actors
Recruiter, HR Manager
Main Flow
1. Recruiter opens "New Offer" form 2. Fills: title, description, contract type, location, experience, salary range, skills, deadline 3. Saves as draft or publishes directly (status=open) 4. System creates offer, logs action in audit trail 5. If open: visible to authenticated candidates immediately
Exceptions
Missing required fields → 400 with field-level errors. Unauthorized role → 403.
UC-05Offer Status Lifecycle Recruiter / HR▼
Flow
draft → open → in_progress → closed (no reversal allowed after closed)
Side Effects
When closed: new applications automatically rejected. Pending applications receive status-change email.
Exceptions
Invalid transition → 400. Closed → open not allowed.

Applications

UC-06Submit Application Candidate▼
Preconditions
Candidate logged in. Offer has status=open. Not already applied.
Main Flow
1. Candidate selects offer and clicks Apply 2. Uploads CV (PDF, max 5MB) + optional cover letter 3. System creates application with status=pending 4. Celery task queued: parse CV with AI engine 5. Notification sent to recruiter (in-app + email) 6. Confirmation email sent to candidate
Exceptions
Already applied → 400 with clear message. Offer closed → 403. File >5MB → 400.
UC-07Update Application Status Recruiter / HR▼
Flow
pending → in_review → shortlisted → rejected | hired
Side Effects
Every status change triggers a Celery task that sends a notification email to the candidate. Action logged in audit trail.

Interviews

UC-08Schedule Interview Recruiter▼
Preconditions
Application is shortlisted.
Main Flow
1. Recruiter fills: type (phone/video/onsite/technical), date/time, duration, location or link 2. System creates interview with status=scheduled 3. Celery beat schedules reminder 24h before scheduled_at 4. Email notifications sent to both candidate and recruiter
Exceptions
Past date → 400. Scheduling conflict → 400 with available slots hint.
UC-09Conduct & Evaluate Interview Recruiter▼
Main Flow
1. Recruiter opens interview session page, takes real-time notes (auto-saved every 30s) 2. After interview: fills structured evaluation form • Technical score (1–5) • Communication score (1–5) • Motivation score (1–5) • Problem solving score (1–5) • Recommendation: hire | reject | hold | next_round • Written comments 3. System calculates overall_score (weighted average) 4. Interview status → completed
Exceptions
Already evaluated → 400. Score out of 1–5 range → validation error.
UC-10Automated Interview Reminder System / Celery▼
Trigger
Celery beat runs every hour. Finds interviews scheduled in next 24h with reminder_sent=False.
Action
Sends email to candidate and recruiter. Sets reminder_sent=True to prevent duplicates.
Exceptions
Email delivery failure → logged, task retried up to 3 times with exponential backoff.

AI Engine (Phase 2)

UC-11AI CV Parsing System / AI▼
Trigger
Celery task triggered automatically on application submit.
Process
spaCy NER extracts: name, email, skills, education, certifications, experience years. Detects inconsistencies. Normalises skill names (Python 3 → Python). Saves to application.ai_parsed_data.
Exceptions
Unsupported format → logged with error flag. Parse failure → retried 3x.
UC-12AI Candidate Scoring System / AI▼
Process
1. skills_match = intersection(CV skills, offer required skills) 2. experience_match = CV years vs offer minimum 3. keyword_score = TF-IDF cosine similarity (CV text vs offer description) 4. match_score = weighted sum of above scores 5. Saves ai_engine_candidatescore record
Output
Score 0–100 visible to recruiter in application list. Used for automated ranking.
UC-13Generate PDF Debrief Report HR / Recruiter▼
Preconditions
At least 2 interviews completed for the offer. Evaluations exist.
Process
WeasyPrint renders PDF with: ranked candidate table, score breakdown, evaluation comments, recommendation summary. Triggered as Celery task.
Output
PDF download. Also stored in reports/ for future access.

Senior Developer Rules

These rules are non-negotiable

A violation in one area cascades into bugs, security holes, or an unmaintainable codebase. No exceptions in code review.

Backend — Django / DRF

No hardcoded values — all config via django-environ from .env
Custom User model defined BEFORE the first migration — impossible to change later
Views are thin — all business logic lives in services/ layer only
Never block HTTP requests — emails, AI, reports go to Celery tasks
Always use select_related() and prefetch_related() — zero N+1 queries
Validate at serializer layer, never in views
All endpoints versioned: /api/v1/ — never expose unversioned API
Use django-filter for all filtering — no raw query params in views
All exceptions return consistent JSON: {error, code, detail}
Migration files never edited manually after commit
Never use raw SQL unless parameterised — always use the ORM
File uploads go to object storage — never Django media/ in prod

Frontend — React

One Axios instance — never instantiate axios directly in a component
JWT refresh handled in interceptor — invisible to the user
Feature-sliced — features never import directly from each other
All form validation uses Zod schemas — no manual error state
Server state = React Query — no useEffect for data fetching
Global UI state = Zustand — no prop drilling, no Redux
All API URLs are constants in src/api/endpoints.js
Role-based routing enforced in ProtectedRoute.jsx
Never store tokens in localStorage — memory or httpOnly cookies only
All env variables prefixed with VITE_ in .env

Security Checklist

AreaRequirement
AuthJWT access 15min, refresh 7 days. Auto-refresh transparent to user.
PasswordsPBKDF2 hashing — never plain text or md5 in production
CORSOnly allow frontend origin in CORS_ALLOWED_ORIGINS
Rate LimitingMax 5 login attempts per minute per IP — django-ratelimit
File UploadsValidate MIME type + extension. Max 5MB. Reject non-PDF CVs.
SecretsAll in .env — never in code, git, or logs
SQL InjectionImpossible via ORM. Raw SQL forbidden unless parameterised.
XSSReact escapes by default. Never use dangerouslySetInnerHTML.
HTTPSEnforced in production. HSTS header enabled.

UX Design Principles

Speed

Every page load under 1.5s. Every action gives feedback in <200ms. Skeleton loaders, never blank pages.

Clarity

Every button, label, and message is unambiguous. No jargon. Error messages tell users what to do, not what went wrong.

Forgiveness

Users can cancel, undo, and recover. Destructive actions require confirmation. Drafts auto-saved.

Consistency

Same action looks and behaves the same everywhere. Component library enforces uniformity.

Feedback

Every async action shows loading → then success or error. Buttons disabled during requests. Toast notifications auto-dismiss in 4s.

Role-awareness

Each role sees only what is relevant to them. Navigation, dashboards, and actions are tailored per role.

Role-Specific Dashboards

RoleDashboard Content
AdminUser management, system stats, audit logs, all reports, configuration panel
HR ManagerOffer pipeline overview, interview calendar, candidate funnel stats, export reports
RecruiterMy offers, pending applications with AI scores, today's interviews, evaluation queue
CandidateBrowse open jobs, my applications with statuses, upcoming interviews, notifications

Form Rules

  • Inline validation — errors appear on blur, not only on submit
  • Required fields marked — optional fields explicitly labeled "(optional)"
  • File upload shows file name, size, and a remove button after selection
  • Long forms split into wizard steps — not one infinite scroll page
  • On submit: button shows "Saving..." → on success route to result → on error stay on form with error highlighted
  • Password fields have a show/hide toggle
  • All date pickers use a calendar component — no raw text input for dates

Follow this exact order. Each phase builds on the previous. Never skip a step — the foundation must be solid before adding features.

Configure DATABASE_URL. Configure SimpleJWT: access=15min, refresh=7d. Custom token payload includes role. Add /api/v1/auth/ URLs. Test login/refresh/logout.
PHASE 1Core UtilitiesDay 4
Write core/permissions.py (IsRecruiter, IsHRManager, IsCandidate). Write core/exceptions.py (custom_exception_handler). Write core/pagination.py. Write core/mixins.py (AuditMixin).
PHASE 2Jobs AppWeek 2
Models: Skill, JobOffer. Serializers with nested skills. ViewSet with filter by status/type. Permissions: recruiter/HR can write, candidates read-only. Tests: CRUD + full permission matrix.
PHASE 2Applications AppWeek 2
Model: Application with status lifecycle. File upload for CV (PDF only, 5MB limit). Unique constraint (candidate, offer). Service: transition logic. Celery task: send_status_email.
PHASE 2Interviews AppWeek 3

PHASE 3React + Vite SetupWeek 3

PHASE 3Feature ModulesWeek 4
Build features in order: jobs → applications → interviews → candidates. Each: api.js + hooks/ + components/. Shared DataTable with pagination and filters.
PHASE 4AI Engine (Scaffold)Week 5
Create ai_engine app. Skeleton services: cv_parser.py, matcher.py. Celery tasks that call services. Models: CandidateScore. API endpoints to trigger and retrieve scores.
PHASE 4Reports ModuleWeek 5
WeasyPrint PDF generation. openpyxl Excel export. API endpoints with role permissions. Frontend: Export buttons on dashboard. File download from API response.
PHASE 5TestingWeek 6
pytest-django + factory_boy for all backend apps. Target 80%+ on services. Vitest + RTL for frontend. MSW for API mocking. Test all permission scenarios.
PHASE 6Production HardeningWeek 7
Production settings: DEBUG=False, HTTPS, HSTS, CORS locked. GitHub Actions CI: lint → test → build. drf-spectacular OpenAPI docs. README with full setup guide.
